# 编程 Agent 如何重塑工程、产品和设计

## 摘要

Harrison Chase（[[harrison-chase|LangChain CEO]]）分析编程 Agent 对软件公司 EPD（工程/产品/设计）三大职能的系统性影响。核心论点：编程 Agent 让代码实现成本趋近于零，**瓶颈从实现转向评审**，催生 Builder/Reviewer 二分角色，通才价值暴涨，系统思维成为核心差异化能力。PRD 的传统流程（PRD→设计稿→代码）已死，但描述意图的文字仍是原型的必备伴侣——"未来的 PRD 可能就是结构化的、带版本管理的 prompt"。

## 关键要点

### 核心论点：瓶颈转移

```
旧世界：想法 → PRD → 设计稿 → 代码（实现是瓶颈）
新世界：想法 → 原型 → 评审（评审是瓶颈）
```

### PRD 已死 → PRD 万岁

1. **PRD 已死**：传统 PRD→设计稿→代码 流程终结。编程 Agent 能将想法直接变成可运行原型
2. **瓶颈转向评审**：代码生成成本极低 → 原型大量涌现 → 工程评审架构、产品评审需求、设计评审易用性
3. **PRD 万岁**：描述意图的文字仍不可或缺。评审原型时需知道"哪里是有意为之还是偶然写成"。未来的 PRD = 结构化的、带版本管理的 prompt

### Builder vs Reviewer 二分角色

| 角色 | 能力要求 | 工作方式 |
|------|----------|----------|
| Builder | 产品思维 + 编程 Agent + 基本设计直觉 | 小功能独立完成，大功能做原型 |
| Reviewer | 顶尖系统思维 + 极快评审速度 + 强沟通 | 大型复杂功能的深度评审 |

### 七个关键判断

1. **通才比以往更有价值**：一人跨产品/设计/工程，省掉沟通开销，只和 Agent 沟通
2. **使用编程 Agent 是必选项**：不用就被用的人替代
3. **好的 PM 更好，差的 PM 更差**：差想法现在带原型出现，浪费更多评审时间 + 上线惯性更大
4. **每个人都需要产品意识**：没有产品意识就制造待评审垃圾
5. **专业化门槛更高**：不仅要在领域出色，还要快、会沟通
6. **系统思维是核心技能**：实现成本 ↓ → 方向正确性 ↑ 重要性
7. **每个人都觉得自己的角色受益最大——而且都没说错**：全才可以来自产品、设计或工程任一方向

### 系统思维的三领域心智模型

- **工程**：对架构、API、数据库设计有清晰心智模型
- **产品**：对用户真正需要什么（而非嘴上说想要什么）有清晰心智模型
- **设计**：对为什么某个东西"感觉对"有清晰心智模型

## 重要引用

> "编程 Agent 颠覆了这一切。编程 Agent 能把一个想法直接变成可运行的软件。"

> "沟通是所有事情中最难的部分，它拖慢一切。一个人如果能同时搞定产品、设计和工程，就比三个人的团队快，因为省掉了沟通开销。"

> "一个对现有产品有直觉般理解的人——知道哪里是短板、哪里是亮点、以及如何迭代让产品变得更锋利。这种人最稀有的版本，站在文化与深度技术的交叉点上。"

## 前提与边界

- **前提假设**：Chase 对编程 Agent 影响 EPD 的分析基于 LangChain 的客户观察和行业案例，代表了 AI 基础设施公司 CEO 的视角，存在利益相关性（LangChain 是 Agent 工具提供商）
- **数据可靠性**：Chase 的个人观点 + 行业案例观察，非系统性调研数据；七个关键判断为定性推断，缺乏定量支撑
- **不适用场景**："PRD = 结构化 prompt"的论点在大企业中可能面临合规阻力（审计追踪、签名审批流程）；"通才价值暴涨"的判断假设编程 Agent 已达到足够的代码质量水平，对安全关键或高合规行业（金融、医疗、航空）尚不成立

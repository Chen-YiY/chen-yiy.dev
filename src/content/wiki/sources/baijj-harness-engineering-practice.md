# 万字干货！Harness Engineering 如何工程化落地？（白家杰, 2026-04-22）

## 摘要

28,000 字的实战工程文档。作者白家杰以 JK Launcher（Unity 研发流程桌面启动器）为真实案例，从零到一完整拆解了 Harness Engineering 的落地历程。**首个按时间顺序、包含失败和迭代的全量落地记录**。核心贡献：四块拼图模型（约束与流程 + 反馈 + 知识库 + 进化）、Rule → Skill → Scripts 渐进下沉路径、结构化调度胜出去中心化协作的实战 PK、七 Agent 角色体系、总验证脚本作为反馈闭环。

## 关键要点

- **四块拼图模型**：约束与流程（Rule/Skill/多 Agent/Workflow）+ 反馈（总验证脚本）+ 知识库（dev-map/任务看板）+ 进化（规范沉淀、人与 AI 分工）。四块相辅相成，缺一块系统就失衡
- **Rule → Skill → Scripts 渐进下沉**：Rule 只能做"原则约束"不能做"流程执行"。高频固定动作做成 Skill 标准化，可判定的约束下沉为脚本门禁。自然语言约束永远存在解释空间，脚本是最终裁判
- **三种多 Agent 路线实战 PK**：① 单 Agent 强化（成本低但角色冲突严重）→ ② 去中心化协作（灵活但路径不稳定、难维护）→ ③ **结构化调度**（流程清晰、可审计、可维护）胜出。"真正贵的不是 token，真正贵的是失控"
- **七 Agent 角色体系**：PM（路由+进度）/ 需求分析 / 方案设计 / 闸门总控（可行性分析）/ 开发实现 / 代码审查 / 测试验证。不是拍脑袋拆的，是被问题一步步逼出来的。PM = 总路由器不是总专家
- **总验证脚本 = 反馈闭环**："AI 说做完了" → "脚本判定你通过了才算"。覆盖静态规范 + 编译 + 测试 + 工程一致性。是 Harness 从"软约束"走向"硬验收"的决定性环节
- **基线对比机制**：开发前跑一次，开发后跑一次，前后对比防止 AI 把历史遗留问题当借口
- **角色契约**：每个 Agent 有输入输出边界定义（必须读什么、必须写什么、什么情况阻塞、什么情况交回 PM）。角色边界不靠人记住，而靠契约文件维持
- **dev-map + 任务看板**：项目级索引而非百科全书。开发导航图（开发 Agent 维护）+ 任务总览（PM 维护）。谁动代码谁改地图
- **Memory 在团队级 Harness 中应靠边站**：个人偏好可用 Memory，但团队对齐的东西必须进仓库（SPEC/Rule/Scripts/dev-map）。"档案进仓库，故事进测试，手册进脚本"
- **最小起步路径（7 步）**：SPEC → 关键 Rule → 高频 Skill → 单 Agent 失稳时拆多 Agent → 复杂时补流程定义和角色契约 → 持续迭代时补 dev-map 和任务看板 → 想推闭环时再考虑 MCP
- **人与 AI 的关系**：人从执行者上移为系统设计者和最后责任人，AI 在制度里高强度执行。"AI 不是助手，更像是一支执行力极强但必须被制度化管理的团队"

## 重要引用

> "Harness Engineering 真正解决的不是'怎么让 AI 更聪明'，而是怎么把 AI 从一个会临场发挥的模型，变成一个在工程里可约束、可协作、可校验、可持续维护的执行系统。"

> "Rule 不是没用，而是 Rule 只能做'原则约束'，不能做'流程执行'。"

> "真正贵的不是 token，真正贵的是失控。"

> "下游不能直接改上游文档。当下游认为上游产出不合格时，只能提出阻塞项，由 PM 把流程正式打回给上游。"

> "总验证脚本不再是'建议你验证一下'，而是在客观上决定：这次开发到底算不算完成。"

> "PM 不是流程中央的'总专家'，而是流程中央的'总路由器'。"

> "事实像档案，例子像故事，规矩像手册——手册应进仓库，别只靠聊天记忆。"

## 前提与边界

- **前提假设**：全部结论以 JK Launcher（WPF + .NET 桌面应用）为唯一实战案例，这是一个"有明确软件工程流程"的开发项目，结论的可推广性受限于项目类型
- **数据可靠性**：28,000 字详细实战记录，过程透明度高，但属于单一项目经验；三种多 Agent 路线的 PK 结果（结构化调度胜出）基于该特定项目的需求特征，不能直接推广为普遍规律
- **不适用场景**：七 Agent 角色体系按软件工程流程分工（PM/需求分析/方案设计/开发/审查/测试），不适用于探索性或研究性任务（目标不明确、无固定流程可遵循）；"结构化调度 > 去中心化协作"的结论在开放式问题、创意生成等场景中可能不成立——去中心化协作在这些场景中可能更优

# RAG 与编译式知识库的对比

## 定义

两种利用 LLM 处理文档集合的方式：**RAG**（Retrieval-Augmented Generation）在查询时即时检索原始文档片段；**编译式知识库**（Compiled Wiki）让 LLM 预先将知识编译成持久、互联的结构化页面。

## 前提与边界

**前提假设**：
- 两者解决的是**同一类问题**：让 LLM 获取外部知识以生成更准确的回答。对比才有意义的前提是它们可以被用于同一个使用场景
- 编译式方案的前提是知识有**重复使用价值**——编译一次、查询多次，前期投入可通过后续查询摊销
- RAG 的前提是知识库规模大到无法全部预编译，或文档更新频率高到编译跟不上变化

**适用边界**：
- **具体优劣取决于查询模式和知识库规模**。一次性查询（如分析一份新文档）天然适合 RAG；重复性查询（如持续运营的知识库）天然适合编译式
- RAG 在**大规模文档集合**（数千到数万篇）下仍具优势——编译式的交叉引用维护成本随规模非线性增长，`index.md` 可能不够用
- 编译式在**中等规模**（~100 源，数百页 wiki）下优势最大，超过此规模可能需要引入 RAG 作为补充检索层（混合模式）
- RAG 无法解决**跨文档矛盾检测**和**累积性洞察**的问题；编译式无法解决**实时性**和**零成本冷启动**的问题——两者在信息时效性和知识积累性上各有短板
- 编译式存在**"错误复利"风险**：编译时引入的错误会持久化在 wiki 中，与正确知识一样被反复引用，检测和纠正成本可能高于 RAG 的"每次从零开始"

## 深入解释

### RAG 模式

工作方式：用户上传文件 → 建立 embedding 索引 → 查询时检索相关片段 → LLM 基于片段生成答案

典型代表：NotebookLM、ChatGPT 文件上传、大多数 RAG 系统

核心特征：
- 每次提问都从原始文档**重新发现**知识
- 没有积累——综合五份文档的细微问题，LLM 每次都要重新拼凑
- 查询间无状态，之前查询的洞察不会保留

### 编译式 Wiki 模式

工作方式：投喂源材料 → LLM 阅读、提取、整合到 wiki → 查询时搜索已编译知识

核心特征：
- 知识编译一次，持续更新
- 交叉引用已存在，矛盾已标记，合成已反映所有已读内容
- wiki 是**持久复利资产**——每新增一个源或问题，它都变得更丰富
- 好的查询答案可回写为新的 wiki 页面，探索本身也产生复利

### 关键差异

| 维度 | RAG | 编译式 Wiki |
|------|-----|-------------|
| 知识存在形式 | 原始文档 + embedding 索引 | 已编译的互联 Markdown 页面 |
| 查询成本 | 每次重新检索+合成 | 搜索已编译页面（更快） |
| 积累性 | 无——每次从零开始 | 强——知识持续复利 |
| 维护 | 仅维护索引 | LLM 持续更新所有相关页面 |
| 矛盾处理 | 每次可能得出不同结论 | 矛盾已标记和记录 |
| 适用规模 | 大规模文档集合 | 中等规模（~100 源，数百页） |
| 基础设施 | 需要 embedding 模型和向量数据库 | 仅需 Markdown 文件和索引 |

## Anthropic 的实战验证

> [!important] Anthropic 亲身验证
> 2026-04-10，Anthropic Claude Code 团队公开发文承认**从 RAG 转向 Agent 自主搜索**。原因：RAG 让 Claude 被**给**上下文而非**自己发现**上下文；改为给 Claude Grep 工具后，Agent 越聪明越擅长自己构建上下文。这与 Karpathy 的"编译 > 检索"论点完全一致。
> — 详见 [[seeing-like-an-agent]]

## 生产级失败数据（饼干哥哥 2.0→3.0）

[[biscuitbrother|饼干哥哥]]的 2.0 系统（Runtime RAG）在 200+ 篇资料规模下的量化失败：

| 问题 | 数量 | 根因 |
|------|------|------|
| 概念定义冲突 | 37 处 | 无编译层，每次查询独立拼凑 |
| 事实矛盾 | 4 处 | 查询间无状态，无法标记矛盾 |
| 从未被引用的文章 | 60+ 篇 | 无交叉引用网络，收藏即遗忘 |
| 同一问题两次不同答案 | 持续发生 | 每次从零检索，命中不同片段 |

> "同一个问题问 AI 两次，答案不一样。创作链路是自动化了，但底层的知识库还是一个信息垃圾场。2.0 的天花板在这里。"

**2.0→3.0 的命名框架**：
- **2.0 = Runtime RAG**：提问时才翻原始文档（NotebookLM、ChatGPT 文件上传的逻辑）
- **3.0 = Compile-time Wiki**：LLM 预先将知识编译为持久、互联的结构化资产

[[biscuitbrother|饼干哥哥]]的商业验证：3.0 系统管理 4 个公众号矩阵，一篇 AI 全流程文章带来 20 万销售额。核心不是方法论先进，而是**维护成本终于低到可以忽略**。

## 三阶段演进（@shannholmberg）

[[shannholmberg-ai-knowledge-layer|Shann Holmberg]]将 RAG→编译式的演进系统化为三阶段：

| 阶段 | 时期 | 模式 |
|------|------|------|
| 1 | 2020-2023 | 一次性 RAG（分块 + 搜索） |
| 2 | 2023-2024 | Agentic RAG（多跳检索） |
| 3 | 2025+ | Context Engineering（Agent 自建上下文） |

新增量化证据：**Graphify 测量到编译式方案每次查询节省 71.5 倍 token**（vs 搜索原始文件）。与 Karpathy 的"~100 篇文章时编译优于 RAG"和饼干哥哥的"37 处冲突 + 60+ 篇未引用"形成三角量化验证。

# LLM Wiki 模式

## 定义

一种用 LLM Agent 增量构建和维护持久知识库的模式。区别于 RAG 在查询时从原始文档即时检索，LLM Wiki 让 LLM 持续编译知识——将源材料提取、整合到结构化的、互联的 Markdown 文件集合中，并在新信息到来时保持更新。

## 前提与边界

**前提**：
- Karpathy 的原始方案假设 LLM 能在单次操作中准确修改多个文件且不出错——实际中模型偶尔会遗漏交叉引用或格式不一致，需要 Lint 流程兜底
- 假设 wiki 规模在几百页以内，此时 `index.md` 作为人工可维护的全量索引足以替代向量搜索，Agent 通过"读 index → 定位页面 → 读页面"即可高效检索
- 编译式优于检索式的前提是 LLM 的编译质量（摘要、提取、整合）足够高，且编译成本（token 消耗）在可接受范围内

**边界**：
- 大规模下（数千页以上）index.md 的线性浏览策略将失效，可能需要引入向量搜索或分层索引结构
- 三步编译法（浓缩→质疑→对标）的"质疑"步骤依赖 LLM 的批判性推理能力，较弱的模型可能产出浅层质疑而非深度前提检验
- 饼干哥哥的生产级验证基于公众号矩阵场景（商业内容研究），其他知识领域（如代码库、科研论文）的适用性有待独立验证
- 本 wiki 自身仅约 30 页规模，尚未触及规模瓶颈

## 深入解释

LLM Wiki 的核心架构分三层：

1. **Raw sources**（不可变层）— 原始文档，LLM 只读不写。这是事实来源
2. **The wiki**（编译层）— LLM 生成的 Markdown 文件集：摘要、实体页、概念页、比较、概览。LLM 完全拥有此层
3. **The schema**（规则层）— 如 `CLAUDE.md` 或 `AGENTS.md`，告诉 LLM 如何组织 wiki、遵循什么约定、执行什么工作流

三大核心操作：
- **Ingest**：投喂新源 → 阅读、提取、整合（单源可触及 10-15 个 wiki 页面）
- **Query**：提问 → 搜索已编译知识，合成答案。好的答案可回写为新的 wiki 页面
- **Lint**：健康检查 → 矛盾、过时、孤儿、缺口检测

两个关键导航文件：
- **index.md**：内容目录，按类别索引所有页面。中等规模下足以替代 embedding RAG
- **log.md**：append-only 时间线日志，记录所有操作

## 实际应用

- **个人知识管理**：追踪目标、健康、心理学，构建关于自己的结构化图景
- **深度研究**：数周/月内阅读论文，构建递进式综合
- **读书笔记**：逐章归档，构建角色/主题网络
- **团队知识库**：内部 wiki，由 Slack/会议/项目文档自动维护
- **竞品分析、尽调、旅行规划、课程笔记**

## 三步编译法（饼干哥哥改进）

[[biscuitbrother|饼干哥哥]]在 Karpathy 原始编译流程基础上，发现了一个关键盲区：**纯摘要只压缩信息，不生成新知识**。两篇观点相反的文章，摘要可能相同。为此增加了"质疑"和"对标"两步：

### 第一步：浓缩

剃刀法则：删掉这条信息会影响理解吗？不会就删。输出核心结论（≤3 条）+ 支撑每条结论的关键证据。

### 第二步：质疑（关键差异）

针对每条核心结论：
1. 这个结论依赖哪些前提假设？
2. 如果前提不成立（换行业/换市场/换规模），结论还成立吗？
3. 数据来源可靠吗？样本量、时间范围、地域限制？
4. 有没有作者没提到的反例或边界条件？

> **为什么这步重要**：编译一篇"用 GPT-4 做跨境电商选品，转化率提升 40%"的文章时，AI 指出核心前提是"目标市场是北美"——如果做东南亚市场，40% 可能完全不成立。否则这个数字就作为孤立事实躺在 Wiki 里误导你。

### 第三步：对标（跨域综合）

1. 其他领域有没有类似现象？（技术↔商业↔管理↔科学）
2. 知识可以迁移到哪些场景？
3. 如果有跨域关联，创建或更新对应概念条目

> **生产级验证**：编译"AI Agent 多步推理链第 5 步后失控"文章时，AI 对标发现这跟"供应链超 4 环节出错率指数上升"是同一现象——**复杂链条的可靠性随环节数指数下降**。这个跨域洞察被写入概念页，同时链接到 AI Agent 和供应链两个领域。在两个领域分别工作多年的人，从未意识到这是同一种现象。

### 与本 Wiki 的对映

本 wiki 的 Ingest 流程已隐含"质疑"（通过 `[!contradiction]` callout 标记矛盾）和"对标"（通过 `wiki/insights/` 跨源综合）。饼干哥哥的三步法将隐含实践显式化，提供了一个可操作的编译模板。

## 生产级验证

[[biscuitbrother|饼干哥哥]]的 2.0→3.0 升级提供了此模式的首个公开商业验证：

- **规模**：4 个公众号矩阵、200+ 篇研究资料、35 个 AI 技能包
- **商业结果**：一篇 AI 全流程文章带来 20 万销售额
- **2.0 的失败数据**：37 处概念定义冲突、4 处事实矛盾、60+ 篇从未引用——量化了 RAG 模式的无积累性
- **核心洞察**：3.0 能持续运转的根本原因不是方法论先进，而是**维护成本终于低到可以忽略**——LLM 一次操作同时修改 15 个文件，不会忘不会烦

## 健康检查四维度协议

[[biscuitbrother|饼干哥哥]]提供的可操作健康检查框架（对 WIKI.md 中 Lint 流程的细化）：

| 维度 | 检查内容 |
|------|----------|
| 一致性 | 同一概念在不同条目中的定义是否一致 |
| 完整性 | 每个条目是否包含所有必填字段 |
| 孤岛检测 | 入链和出链都少于 2 的页面——需要建立关联或考虑合并 |
| 跨目录一致性 | 多账号/多项目场景下风格规则是否遗漏或冲突 |

## 亦仁的创业者实践

[[yiren|亦仁]]（生财有术创始人）的 Obsidian + Claude 工作流提供了 LLM Wiki 模式在创业者场景的完整实例：

- **工作流反转**：从"Obsidian 往 Claude 走"变成"Claude 往 Obsidian 走"——与 LLM Wiki 的"Cowork 往 wiki 走"完全一致
- **数据主权**：亦仁强调"不要和 Claude 产生感情"，所有数据沉淀到本地 Obsidian + iCloud。这与 LLM Wiki 的 raw/ 不可变层理念一脉相承：**知识资产不能依赖任何单一平台**
- **知识复利**：Notebook、Obsidian、飞书知识库 = "自己的个人 LLM"。与 LLM Wiki 的 wiki/ 编译层目标一致——构建越用越懂你的持久知识库
- **信息管理**：微信群聊/公众号/YouTube → NotebookLM 提炼 → Obsidian 沉淀，是 LLM Wiki Ingest 流程的变体

## 进化：AI Knowledge Layer（@shannholmberg）

[[shannholmberg-ai-knowledge-layer|Shann Holmberg]]在 LLM Wiki 基础上提出 **AI Knowledge Layer 双层架构**：

- **KBL（知识库层）** = 本 wiki 的 wiki/ 层（动态、Agent 维护）
- **BF（品牌基础层）** = 新增的静态锚点层（语体、风格、品牌规则，仅人可改）

核心贡献：将"语体/风格/品牌"从 Schema（LLM 行为规则）中独立出来，成为 Agent 不可修改的锚点。这解决了 Agent 自主进化时偏离品牌一致性的问题。

新增量化证据：Graphify 测量到编译式方案每次查询节省 **71.5 倍 token**（vs 搜索原始文件）。

详见 [[ai-knowledge-layer]]。
